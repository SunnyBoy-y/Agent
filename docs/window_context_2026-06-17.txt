   用户截图指出 chat1 回复里重复了用户问题开头，例如用户说“今天心里有点闷，想聊聊天”，模型回复也先重复这句话。

   用户要求通过提示词注入让模型感知当前时间、指导现在几点，但时间不进入记忆体系，避免污染上下文。

3. 参考 Hermes 改提示词规范
   用户要求参考 C:\Users\13600\Desktop\PR\hermes-agent 的提示词规范，保证模型不只是简单回复，而是张弛有度、像真人、记得过去发生的事情、适当提醒。

4. chat1 分层记忆
   用户要求 chat1 每次传给模型的记忆做分层:
   - 最近 5 轮滑动窗口保留原文。
   - 超出的部分补充上文摘要。
   - 每次给模型的记忆参考 Hermes 编排，确保不忘事。

5. 后台信息处理、事件库、提醒队列、chat2
   用户进一步提出大需求:
   - 新增后台的信息处理与保存区域。
   - chat1 所有对话数据经过后台子 AI 分析后，整理更新画像、消除记忆歧义。
   - 例如“昨天说‘明天要去看车’”，应解析成明确日期事件: “YYYY-MM-DD，用户 xx 准备去看车”。
   - 新增信息要逐条拆分。
   - 插入近期事件库时检查事件冲突、过期事件。
   - 无法逻辑推理消歧的内容进入提醒队列。
   - 最高优先级提醒嵌入每次给 LLM 的历史背景，标明“恰当语境提醒”。
   - 后台检测到提醒已触发后，提醒事件出队。
   - 提醒队列和事件记录独立存储。
   - 新增 chat2 接口，服务子女端。
   - chat2 输入需要包含当前用户与老人的关系 type，例如“妈妈”“爸爸”等，用于 AI 回复时称呼老人。
   - chat2 维护自己的上下文和记忆，同时可继承 chat1 的部分上下文，继承内容由摘要处理智能体后台处理。
   - chat1 历史背景里必须写清楚老人个人画像。
   - 老人画像来自超出 5 轮滑动窗口后对历史记录摘要过程中的提取。
   - 历史摘要触发条件: 每 5 轮对话触发。
   - 历史摘要后台并行处理，不阻塞 chat1。
   - 摘要成功后才影响下一次上下文；失败或未完成不影响当前/下一轮正常上下文，可以临时超过 5 轮。
   - 摘要提取同时服务于摘要、画像更新、消歧等，可设计多次推理。
   - chat1 长连接优先实时性。done 只表示第一段话结束；若后台/agent 后续推理出工具调用，如显示照片、给子女发悄悄话，可继续推送。agent 链路真正结束才标志完整一轮结束。
   - 打断定义: 第 n 轮尚未完整回复，用户点击打断并输入新一轮内容。新一轮上下文应包含已推送内容，并标记“被打断”。参考 Hermes 打断处理。
   - chat1 服务老人，chat2 服务子女。老人是被照护对象，部分老人数据可共享给子女智能体辅助决策。

六、上一次未完成的大需求建议实现方案
----------------------------------
建议先落一个可运行的 MVP，然后逐步把 deterministic 提取替换为真正的子 AI 多轮推理。

1. 新增服务: src/services/memory_reflection_service.py
   职责:
   - 维护近期事件库 recent_events.json。
   - 维护提醒队列 reminder_queue.json。
   - 维护给 chat2 使用的共享老人上下文 shared_elder_context.json。
   - 后台处理 chat1 每轮对话，不阻塞 chat1 首段回复。

2. 推荐接口
   - process_after_chat_turn(elder_user_id, turn_id, user_input, assistant_response, context=None, now=None)
   - get_prompt_context(elder_user_id, limit=1)
   - get_shared_elder_context(elder_user_id)
   - mark_triggered_reminders(elder_user_id, text, turn_id)

3. 事件提取 MVP
   - 用规则/正则先支持:
     - 今天、明天、后天、昨天
     - 下周一到周日
     - X月X日
     - 关键词: 看车、体检、看医生、复诊、买菜、吃药、打电话、见朋友等
   - 相对日期以当前北京时间日期为基准解析。
   - 存储 event:
     - id
     - elder_user_id
     - title
     - event_date
     - status: active / expired / conflict / needs_clarification / completed
     - source_turn_id
     - evidence
     - created_at
     - updated_at

4. 冲突/过期处理
   - event_date < today: 标记 expired。
   - 同一天相近 title 或语义相近: 可先简单标记 conflict。
   - 无法解析日期但有明显计划/提醒意图: 进入 reminder_queue，status=needs_clarification。

5. 提醒队列
   - reminder:
     - id
     - elder_user_id
     - event_id
     - title
     - remind_after / event_date
     - priority
     - status: pending / triggered / dismissed
     - guidance: “在恰当语境自然提醒，不要突兀打断”
   - get_prompt_context() 返回最高优先级 pending reminder。
   - chat1 每次构造 memory_context 时嵌入 top reminder。
   - _finalize_realtime_chat_turn() 后判断 user_input + assistant_response 是否已经自然提醒，若触发则出队。

6. chat1 接入点
   - 在 process_chat_stream() 读取 profile/history_layers 后，同步快速读取 memory_reflection_service.get_prompt_context(user_id)。
   - 将 top reminder 写入 chat_context 和 prompt_payload。
   - _build_memory_context() 增加 reminder_context 参数，并渲染为 [pending_reminder]。
   - 在 _finalize_realtime_chat_turn() 保存 add_memory() 后，schedule/调用 memory_reflection_service.process_after_chat_turn()。

7. chat2 接口
   - 新增 Chat2Request，字段建议:
     - elder_user_id: str
     - child_user_id: str
     - type: str = "家人"
     - message: str
     - context: Dict[str, Any] = {}
   - 新增 /api/chat2。
   - chat2 使用 SSE 格式，与 chat1/family chat 保持 data: json。
   - chat2 自己存上下文，例如:
     - users/{elder_user_id}/family/{child_user_id}/chat2_history.json
   - chat2 读取:
     - memory_reflection_service.get_shared_elder_context(elder_user_id)
     - family_context_service.build_family_chat_context(elder_user_id, child_user_id)
   - 回复时用 relation/type 称呼老人，例如“你妈妈”“你爸爸”。
   - 不把 chat2 对话写入 chat1 老人记忆。

8. chat1 done / agent_done
   - 当前 chat1 会发:
     - step realtime_chat running
     - token
     - step realtime_chat done
     - done stop
   - 用户新需求提出 done 只代表第一段话结束，后续工具/后台动作可继续推送。
   - MVP 可先新增更明确事件:
     - first_response_done
     - agent_done
   - 为兼容前端，旧 done 事件要谨慎调整。可保留 done=stop，同时新增 agent_done；或先只在后台状态里记录 agent_done，避免破坏现有 UI。

9. 打断 MVP
   - active_chat_turns: dict[user_id] -> state
   - state:
     - turn_id
     - streamed_text
     - first_response_done
     - agent_done
     - interrupted
   - process_chat_stream() 开始时检查 context:
     - interrupt_previous_turn_id
     - interrupted_turn_id
     - interrupted=true
   - 若命中活跃未 agent_done 的旧 turn:
     - 标记旧 turn interrupted。
